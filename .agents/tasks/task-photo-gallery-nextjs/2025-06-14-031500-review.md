# Photo gallery with Supabase-backed admin and Pinterest masonry layout

A Next.js 16 App Router application that replaces a static HTML/CSS/JS photo gallery with a full-stack implementation: public-facing home page with hero section and Pinterest-style masonry album grid, dynamic album detail pages at `/{slug}`, and an admin panel with hardcoded credentials for album CRUD and photo upload to Supabase Storage. The approach uses server components for public pages (querying Supabase directly at render time) and client components for the admin panel (calling API routes that proxy to Supabase). Authentication is cookie-based with a static `"authenticated"` token checked in middleware.

**Watch for:** The session cookie is a static string with no cryptographic integrity — anyone who sets `admin_session=authenticated` in their browser has full admin access (confirmed). API routes for album/photo mutation have no auth check, so they're callable without a valid session (confirmed). The `.env.local` file is committed to git with placeholder credentials visible in the repository (confirmed).

## High-level view

The public gallery uses Next.js server components that query Supabase at request time via the anon key, rendering albums in a CSS multi-column masonry layout and individual album photos in the same pattern. The approach is straightforward and avoids client-side data fetching for public pages, which is good for SEO and initial load.

The admin authentication is intentionally simple — hardcoded env var credentials checked against a login API route — but the implementation has a critical gap: the session cookie is a predictable static string rather than a signed or random token, making the auth gate trivially bypassable. The middleware correctly guards `/admin/*` routes, but the API routes under `/api/albums` and `/api/albums/[id]/photos` have no auth middleware, so mutations are publicly callable.

The Supabase integration uses a single client instantiated with the anon key for both server and client contexts. RLS policies in the schema grant public read and authenticated-role write, but since the app never authenticates as a Supabase user (it uses the anon key everywhere), write operations succeed only if Supabase RLS is configured to also allow anon writes — or RLS is effectively bypassed. This creates a mismatch between the stated security model in the SQL schema and the runtime behavior.

Album deletion includes storage cleanup for associated photos, and photo upload includes rollback of the storage object if the database insert fails. These are good defensive patterns for resource management.

## Session security and the auth boundary

The login route in `src/app/api/auth/login/route.ts` validates credentials against env vars and sets a cookie with value `"authenticated"`. The middleware in `src/middleware.ts` checks for this exact string. This means the secret protecting the admin panel is a well-known cookie value, not a cryptographic secret. Any user who discovers or guesses the cookie value — which is a single static English word — gains admin access without credentials.

More critically, the middleware matcher is `["/admin/:path*"]`, which protects only the admin UI pages. The API routes at `/api/albums`, `/api/albums/[id]`, `/api/albums/[id]/photos`, and `/api/albums/[id]/photos/[photoId]` are outside this matcher. A `POST /api/albums` with a JSON body will create an album without any session check. The same applies to photo upload, deletion, and cover assignment. The entire CRUD surface is unprotected at the API layer.

## Supabase client and RLS mismatch

The single Supabase client in `src/lib/supabase.ts` is initialized with `NEXT_PUBLIC_SUPABASE_ANON_KEY`. The SQL schema defines RLS policies that grant write access to the `authenticated` role:

```sql
CREATE POLICY "Allow authenticated insert on albums"
  ON albums FOR INSERT TO authenticated WITH CHECK (true);
```

Since the app never calls `supabase.auth.signIn()` or uses a service-role key, all requests hit Supabase as the `anon` role. Write operations (insert, update, delete) will be rejected by RLS unless the Supabase project has RLS disabled at the table level or has additional policies allowing `anon` writes. The schema as written doesn't grant `anon` any write access, so the app will fail at runtime with permission errors on any mutation.

The fix is either: (a) use the Supabase service-role key in server-side API routes (never exposed to the client), or (b) add RLS policies for `anon` writes (which would make the database publicly writable — not recommended).

## Slug generation and collision handling

Album creation in `POST /api/albums` generates a slug via `slugify(name)` — a deterministic transform. If two albums are created with the same name (or names that normalize to the same slug), the Supabase insert will violate the `UNIQUE` constraint on `albums.slug`. The error bubbles up as a generic 500 with the Supabase error message. There's no retry with a suffix, no user-friendly error, and the slug isn't updated when the album name is edited via PUT (only `name` and `description` are in the update payload).

## Photo upload without server-side validation

The upload handler in `POST /api/albums/[id]/photos` reads a `File` from the form data and immediately uploads it to Supabase Storage. There's no check on file size (a multi-GB upload would be proxied through the Next.js server) and no MIME type validation beyond the client-side `accept="image/*"` attribute. A crafted request can upload arbitrary file types to the storage bucket.

The compensating cleanup — deleting the storage object if the database insert fails — is a good pattern and correctly implemented.

## Public page rendering and image optimization

The home page (`src/app/page.tsx`) is a server component that queries Supabase at render time with no caching directive. Every page load hits the database. For a photo gallery with infrequent updates, adding `revalidate` or `unstable_cache` would reduce load.

The `AlbumCard` component uses `next/image` with `unoptimized` — which opts out of image optimization while still using the component. The album detail page (`[slug]/page.tsx`) uses raw `<img>` tags for photos, missing out on lazy-loading placeholders and responsive sizing that `next/image` provides. This inconsistency between pages means the detail view (which renders the most images) gets the least optimization.

## Committed credentials

The `.env.local` file is committed to the repository with `ADMIN_USERNAME=admin` and `ADMIN_PASSWORD=admin123`. While `.gitignore` lists `.env*` with `!.env.example`, the file was committed in an earlier commit before the gitignore was in place (or was force-added). These credentials are now in git history regardless of future gitignore rules.
